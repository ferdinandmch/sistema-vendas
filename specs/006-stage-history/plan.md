# Implementation Plan: Historico de Movimentacao de Deals

**Branch**: `006-stage-history` | **Date**: 2026-03-29 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-stage-history/spec.md`

## Summary

Expor o historico de movimentacao de stages dos deals via endpoint GET,
permitindo consulta auditavel de todas as transicoes. A feature e
exclusivamente de leitura — nao cria, edita ou deleta registros. Reutiliza
os dados de deal_stage_history ja criados pela feature 004-stage-movement.
Cada registro inclui nomes dos stages de origem e destino via Prisma include.

## Technical Context

**Language/Version**: TypeScript 5.x
**Primary Dependencies**: Next.js App Router, Prisma, Clerk, Zod
**Storage**: PostgreSQL via Prisma (somente leitura nesta feature)
**Testing**: Vitest (contract + integration tests)
**Target Platform**: Server runtime (Node.js)
**Project Type**: Next.js web application
**Performance Goals**: Query de listagem com indice existente
(dealId, changedAt). Include de stage names via relacoes Prisma.
**Constraints**: Somente leitura, ownership isolation via deal.owner_id,
nenhuma mutacao de historico.
**Scale/Scope**: Centenas de deals por usuario, dezenas de movimentacoes por
deal, single-tenant por owner_id.

## Architecture

### Service Layer: `lib/history/history-service.ts`

Novo modulo com uma funcao:

- `listStageHistory(dealId: string, ownerId: string)` — Busca deal com
  ownership, retorna registros de deal_stage_history ordenados por changedAt
  ASC com include de fromStage.name e toStage.name.

Padrao: segue pattern existente do activity-service (find-then-read, error
factories, Prisma include).

### Error Codes

Nenhum erro novo necessario. Reutiliza:
- `DEAL_NOT_FOUND` (404) — deal inexistente ou ownership incorreto
- `UNAUTHORIZED` (401) — sem sessao autenticada

### Route Handler

- `app/api/deals/[id]/history/route.ts` — GET list only

Padrao: segue GET handler de activities (requireAuthenticatedUser, service
call, errorResponse). Sem POST/PUT/DELETE — feature somente leitura.

### Prisma Query Strategy

```
prisma.dealStageHistory.findMany({
  where: { dealId },
  orderBy: { changedAt: "asc" },
  include: {
    fromStage: { select: { id: true, name: true } },
    toStage: { select: { id: true, name: true } },
  },
})
```

Usa include com select para retornar apenas id e name dos stages (nao o
objeto completo). O indice @@index([dealId, changedAt]) ja existe e cobre
a query de listagem ordenada.

### owner_id Resolution

```
const { user } = await requireAuthenticatedUser();
// user.id e o owner_id — usado para buscar deal com ownership
// historico herda ownership do deal
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Historico acessado por outro usuario | Deal buscado com filtro ownerId |
| Historico fora de ordem cronologica | Ordenacao explicita por changedAt ASC |
| Exposicao direta do banco | Encapsulamento via backend + service layer |
| Feature crescer alem do escopo | Edicao, exclusao, analytics fora de escopo |
| Nomes de stages desatualizados | Nomes lidos em tempo real via include |
