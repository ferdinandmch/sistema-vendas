# Implementation Plan: Movimentacao de Deal entre Stages

**Branch**: `004-stage-movement` | **Date**: 2026-03-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-stage-movement/spec.md`

## Summary

Implementar movimentacao transacional de deals entre stages do pipeline. A
operacao atualiza stage_id, stage_updated_at, cria registro obrigatorio em
deal_stage_history e define status (won/lost) quando o stage de destino for
final. Deals finalizados sao imutaveis. Movimentacao para o mesmo stage e
bloqueada.

## Technical Context

**Language/Version**: TypeScript 5.x
**Primary Dependencies**: Next.js App Router, Prisma, Clerk, Zod
**Storage**: PostgreSQL via Prisma (transacoes com `$transaction`)
**Testing**: Vitest (contract + integration tests)
**Target Platform**: Server runtime (Node.js)
**Project Type**: Next.js web application
**Performance Goals**: Operacao transacional simples, sem requisitos especiais
de latencia. Indice em deal_id para queries de historico.
**Constraints**: Transacao atomica obrigatoria (Principio IV), ownership
isolation (owner_id), auditabilidade (deal_stage_history), imutabilidade de
deals finalizados
**Scale/Scope**: Centenas de deals por usuario, dezenas de movimentacoes por
deal, single-tenant por owner_id

## Domain Alignment

- **System Classification**: Auditable sales operating system / state machine.
- **Affected Modules**: Module 1 Core Pipeline (deal state + stage history),
  Module 2 Tracking (deal_stage_history), Module 4 API (route handler + contract)
- **State Transitions**: deal.status: active -> won (via final stage won),
  active -> lost (via final stage lost). Won/lost sao estados terminais —
  nenhuma transicao posterior permitida. deal.stage_id: qualquer stage nao-final
  -> qualquer outro stage (exceto mesmo stage).
- **Ownership Model**: owner_id extraido de AuthenticatedSessionContext.user.id
  via requireAuthenticatedUser(). Deal buscado com filtro ownerId antes de
  qualquer mutacao. Retorna 404 (nao 403) para ownership incorreto.
- **Skills Used**: next-best-practices, postgresql-code-review,
  postgresql-optimization, update-docs

## Architecture

### Service Layer: `lib/deals/deal-service.ts`

Nova funcao adicionada ao service existente:
- `moveDeal(id: string, toStageId: string, ownerId: string)` — busca deal com
  ownership, busca stage destino, valida regras de negocio (same-stage, closed
  deal), executa transacao atomica (update deal + insert history), retorna deal
  atualizado com stage embed.

Padrao: segue pattern existente (find-then-mutate, error factories, Prisma
include). Adiciona uso de `prisma.$transaction()` para atomicidade.

### Validation: `lib/validation/deals.ts`

Novo schema Zod:
- `moveDealSchema` — to_stage_id required (string non-empty)

### Error Codes (novos)

- `SAME_STAGE` (400) — tentativa de mover para o mesmo stage
- `DEAL_ALREADY_CLOSED` (400) — tentativa de mover deal finalizado (won/lost)

### Route Handler

- `app/api/deals/[id]/move/route.ts` — POST move deal

Padrao: segue app/api/deals/[id]/route.ts (requireAuthenticatedUser, Zod parse,
service call, errorResponse).

### Prisma Transaction Strategy

```
prisma.$transaction(async (tx) => {
  // 1. update deal: stage_id, stage_updated_at, status (if final)
  // 2. create DealStageHistory record
  // return updated deal with stage include
})
```

Usa interactive transaction para garantir atomicidade entre update e insert.

### DealStatus Enum Extension

DealStatus: active | won | lost (adicionar won e lost ao enum existente)

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Movimentacao parcial (update sem historico) | Prisma $transaction atomica |
| Historico ausente apos mover stage | Insert dentro da mesma transacao |
| Vazamento de mutacao entre usuarios | Deal buscado com filtro ownerId |
| Status inconsistente em stage final | Status derivado de is_final/final_type no backend |
| Deal finalizado movido indevidamente | Validacao explicita de deal.status != won/lost antes da transacao |
| Movimentacao para mesmo stage criando historico falso | Validacao from_stage_id != to_stage_id |
| Frontend simulando movimentacao | Backend e unica fonte de verdade |
