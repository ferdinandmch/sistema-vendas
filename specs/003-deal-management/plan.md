# Implementation Plan: Gestao de Deals

**Branch**: `003-deal-management` | **Date**: 2026-03-26 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-deal-management/spec.md`

## Summary

Implementar CRUD de deals (criar, listar, detalhe, editar) como entidade central
do pipeline de vendas. Deals vinculam-se a um stage valido e a um usuario
autenticado via owner_id. Backend e fonte unica de verdade para ownership, status
e stage. Respostas incluem dados basicos do stage embutidos.

## Technical Context

**Language/Version**: TypeScript 5.x
**Primary Dependencies**: Next.js App Router, Prisma, Clerk, Zod
**Storage**: PostgreSQL via Prisma
**Testing**: Vitest (contract + integration tests)
**Target Platform**: Server runtime (Node.js)
**Project Type**: Next.js web application
**Performance Goals**: Operacoes CRUD simples, sem requisitos especiais de latencia
**Constraints**: Ownership isolation (owner_id), integridade referencial (stage_id FK),
no hard-delete, validacao centralizada Zod
**Scale/Scope**: Centenas de deals por usuario, single-tenant por owner_id

## Domain Alignment

- **System Classification**: Auditable sales operating system / state machine.
- **Affected Modules**: Module 1 Core Pipeline (deal entity), Module 4 API
  (route handlers + contracts)
- **State Transitions**: Nesta feature, apenas inicializacao: deal nasce com
  status=active. Transicoes de stage e status serao features futuras.
- **Ownership Model**: owner_id extraido de AuthenticatedSessionContext.user.id
  via requireAuthenticatedUser(). Todas as queries de deals filtram por
  ownerId. O frontend nunca controla owner_id.
- **Skills Used**: next-best-practices, postgresql-code-review,
  postgresql-optimization, update-docs

## Architecture

### Service Layer: `lib/deals/deal-service.ts`

Funcoes:
- `listDeals(ownerId: string)` — lista deals do usuario com stage incluido
- `getDeal(id: string, ownerId: string)` — detalhe com ownership check
- `createDeal(data, ownerId: string)` — cria deal com stage validation
- `updateDeal(id: string, data, ownerId: string)` — edita campos permitidos

Padrao: segue stage-service.ts (find-then-mutate, error factories, Prisma include).

### Validation: `lib/validation/deals.ts`

Schemas Zod:
- `createDealSchema` — company_name required, stage_id required, demais opcionais
- `updateDealSchema` — todos opcionais, stage_id/owner_id/status excluidos

### Error Codes (novo)

- `DEAL_NOT_FOUND` (404) — deal inexistente ou ownership incorreto

### Route Handlers

- `app/api/deals/route.ts` — GET list + POST create
- `app/api/deals/[id]/route.ts` — GET detail + PUT edit

Padrao: segue app/api/stages/route.ts (requireAuthenticatedUser, Zod parse,
service call, errorResponse).

### Prisma Include Strategy

Respostas de listagem e detalhe usam `include: { stage: { select: { id, name, position } } }`
para embutir dados basicos do stage. Criacao e edicao tambem retornam com include.

### owner_id Resolution

```
const { user } = await requireAuthenticatedUser();
// user.id e o owner_id para todas as operacoes de deals
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Criacao com stage_id invalido | Service valida existencia do stage antes de criar |
| Vazamento de dados entre usuarios | Todas as queries filtram por ownerId |
| Frontend influenciando ownership | owner_id resolvido exclusivamente no backend |
| Estado inicial inconsistente | status e stage_updated_at definidos no backend |
| Edicao de campos protegidos | Schema Zod aceita apenas campos editaveis |
