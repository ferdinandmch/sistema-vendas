# Implementation Plan: Activities de Deals

**Branch**: `005-deal-activities` | **Date**: 2026-03-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/005-deal-activities/spec.md`

## Summary

Implementar gestao de activities vinculadas a deals, permitindo criacao e
listagem de interacoes comerciais (note, call, meeting, followup). Cada
activity criada atualiza atomicamente deal.last_touch_at. Activities herdam
ownership do deal — nao possuem owner_id proprio. Operacao append-only (sem
edicao ou exclusao).

## Technical Context

**Language/Version**: TypeScript 5.x
**Primary Dependencies**: Next.js App Router, Prisma, Clerk, Zod
**Storage**: PostgreSQL via Prisma (transacoes com `$transaction`)
**Testing**: Vitest (contract + integration tests)
**Target Platform**: Server runtime (Node.js)
**Project Type**: Next.js web application
**Performance Goals**: Operacoes simples de criacao e listagem. Indice em
deal_id + created_at para queries de listagem ordenada.
**Constraints**: Atomicidade obrigatoria na criacao (activity + last_touch_at),
ownership isolation via deal.owner_id, append-only (sem edicao/exclusao),
auditabilidade (Principio II)
**Scale/Scope**: Centenas de deals por usuario, dezenas de activities por deal,
single-tenant por owner_id

## Domain Alignment

- **System Classification**: Auditable sales operating system / state machine.
- **Affected Modules**: Module 2 Tracking (activities como registro de
  interacoes), Module 4 API (route handlers + contracts)
- **State Transitions**: Nenhuma transicao de estado em deal. Activities nao
  alteram deal.status ou deal.stage_id. Unico campo de deal afetado:
  last_touch_at (timestamp de ultima interacao).
- **Ownership Model**: owner_id extraido de AuthenticatedSessionContext.user.id
  via requireAuthenticatedUser(). Deal buscado com filtro ownerId antes de
  qualquer operacao. Activities herdam ownership do deal — sem owner_id proprio.
  Retorna 404 (nao 403) para ownership incorreto.
- **Skills Used**: next-best-practices, postgresql-code-review,
  postgresql-optimization, update-docs

## Architecture

### Service Layer: `lib/activities/activity-service.ts`

Novo modulo com duas funcoes:

- `createActivity(dealId: string, data: CreateActivityInput, ownerId: string)`
  — Busca deal com ownership, executa transacao atomica (create activity +
  update deal.lastTouchAt), retorna activity criada.

- `listActivities(dealId: string, ownerId: string)` — Busca deal com
  ownership, retorna activities ordenadas por createdAt desc.

Padrao: segue pattern existente do deal-service (find-then-mutate, error
factories, Prisma include). Usa `prisma.$transaction()` para atomicidade.

### Validation: `lib/validation/activities.ts`

Novo schema Zod:
- `createActivitySchema` — type obrigatorio (enum: note, call, meeting,
  followup), content opcional (string nullable)

### Error Codes

Nenhum erro novo necessario. Reutiliza:
- `DEAL_NOT_FOUND` (404) — deal inexistente ou ownership incorreto
- `INVALID_REQUEST` (400) — payload invalido (tipo invalido, campos ausentes)
- `UNAUTHORIZED` (401) — sem sessao autenticada

### Route Handler

- `app/api/deals/[id]/activities/route.ts` — GET list + POST create

Padrao: segue app/api/deals/[id]/route.ts (requireAuthenticatedUser, Zod parse,
service call, errorResponse).

### Prisma Transaction Strategy

```
prisma.$transaction(async (tx) => {
  // 1. create Activity (dealId, type, content, createdAt)
  // 2. update Deal: lastTouchAt = now
  // return created activity
})
```

Usa interactive transaction para garantir atomicidade entre create e update.

### ActivityType Enum

ActivityType: note | call | meeting | followup (novo enum no Prisma schema)

### owner_id Resolution

```
const { user } = await requireAuthenticatedUser();
// user.id e o owner_id — usado para buscar deal com ownership
// activity herda ownership do deal
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Activity criada sem last_touch_at atualizado | Prisma $transaction atomica |
| last_touch_at atualizado sem activity | Insert e update na mesma transacao |
| Activity criada em deal de outro usuario | Deal buscado com filtro ownerId |
| Tipo de activity invalido aceito | Zod enum validation no backend |
| Activities listadas sem validar ownership | Deal buscado com ownerId antes de listar |
| Feature crescer alem do escopo | Edicao, exclusao, anexos explicitamente fora de escopo |
| Frontend simulando criacao de activity | Backend e unica fonte de verdade |
